# AdventureWorks scaffolded with EF Core 8

This project contains `AdventureWorksDbContext` and 90 entity classes
generated by `dotnet ef dbcontext scaffold` against the
**AdventureWorks2022** sample database, using **EF Core 8** tooling.

## How it was scaffolded

From the repo root, with `dotnet-ef 8.x` installed globally:

```bash
dotnet ef dbcontext scaffold \
  "Server=CWOLFGAN-03\SQLEXPRESS;Database=AdventureWorks2022;Integrated Security=SSPI;TrustServerCertificate=True" \
  Microsoft.EntityFrameworkCore.SqlServer \
  --project extra-projects/AdventureWorks-EF8/AdventureWorks-EF8.csproj \
  --output-dir Models/Generated \
  --context AdventureWorksDbContext \
  --context-namespace AdventureWorks.Models \
  --namespace AdventureWorks.Models \
  --no-onconfiguring \
  --force
```

The `--use-database-names` and `--data-annotations` flags were
deliberately omitted so the output matches the property-naming and
fluent-configuration style of the existing `AdventureWorks-EF7`
scaffold (PascalCase `BusinessEntityId` rather than DB-column
`BusinessEntityID`, and `OnModelCreating` fluent config rather than
`[Table(Schema="...")]` annotations).

Three Production-schema tables are skipped by EF Core 8's scaffolder
because they use the SQL Server `hierarchyid` / `geography` types
that have no CLR mapping today: `Document`, `ProductDocument`, and
the `OrganizationNode` / `SpatialLocation` columns on `Employee` /
`Address` (the columns are dropped; the parent entities are kept).
Same skips happen on EF Core 9 and 10. EF Core 7 (used for the
existing `AdventureWorks-EF7` scaffold) skipped the same items.

## What still needs to happen to close #41

`tests/Wolfgang.DbContextBuilder-Core-EF8.Tests.Unit/Wolfgang.DbContextBuilder-Core.Tests.Unit-EF8.csproj`
currently `<Compile Include>`s the **EF7-scaffolded** AdventureWorks
files from `../../extra-projects/AdventureWorks-EF7/Models/Generated/`.
The plan was to switch those Include paths to point at
`AdventureWorks-EF8/Models/Generated/` (this project) so the EF8
library is tested against EF8-scaffolded models.

When that rewire is attempted, two architectural mismatches surface:

1. **`DbContextBuilderTestsBase.cs` uses `with`-expression syntax on
   `CountryRegion`** (lines 447, 721 — `expectedCountry with { }` and
   `Select(c => c with { })`). EF8/9/10 scaffolds emit `class`, not
   `record`, so `with` won't compile. The EF7 scaffold uses `record`
   (likely a hand-conversion after the original `dotnet ef` run).

2. **EF8+ scaffolds emit SQL-Server-specific extension calls
   (`HasDefaultValueSql`, `IsClustered`)** that don't resolve from
   the test csproj's package context (the test only references
   `Microsoft.EntityFrameworkCore.Sqlite` transitively). The EF7
   scaffold doesn't use these extensions.

Both need follow-up — either:
- (a) Convert class declarations to `record` post-scaffold, add
  `Microsoft.EntityFrameworkCore.SqlServer` to the test csproj just
  for compile-time resolution of the relational extensions, OR
- (b) Switch the test csproj from `<Compile Include>` to a
  `<ProjectReference>` against this project (then this csproj's own
  SqlServer package handles the extensions and tests get type access
  via assembly reference).

(b) is cleaner architecturally and is the recommended path. It's a
separate stacked PR after this one merges.
